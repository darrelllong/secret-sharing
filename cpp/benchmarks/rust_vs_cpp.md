## Reading the numbers

The C++ port runs 1.25–1.52× faster than Rust on this specific build
(Apple silicon, AppleClang, libc++). **This is not a verdict that
"C++ beats Rust"** — it's a small toolchain-level gap on one build
configuration. On x86 with perf-record, on Linux with jemalloc
replacing the system allocator, or with an unsafe-indexed `BigUint`
(Rust has `get_unchecked` precisely to drop bounds checks in hot
loops), the picture would flip or close. We didn't take those Rust
optimisations because the goal was bit-compatibility, not a
language-vs-language race.

The gap is small enough that it sits within run-to-run noise on the
threshold-scheme bench — the 1.52× peak on `shamir_split_4kb` shrinks
to ~1.3× on consecutive runs and the per-element ratios drift inside
their CIs. Treat the numbers as "consistently nudging C++'s direction
on this hardware/build," not as a stable performance gap.

## Where the (small) gap actually comes from

Both implementations share the same algorithms — schoolbook with the
same Karatsuba threshold, identical Mersenne-127 fold, identical
Lagrange interpolation. The compat tests
(`compat.shamir_share_bytes_match_rust`,
`compat.shamir_reconstruct_from_rust_shares`,
`compat.per_prime_field_mul_matches_rust`) prove the algorithms
produce bit-identical outputs. So the difference is in *how* the same
algorithm gets compiled and what runtime checks ride along.

In rough order of likely contribution on this build:

1. **Bounds checks on `Vec` indexing.** Rust's `vec[i]` panics on
   out-of-bounds via a runtime check the compiler often can't elide;
   the schoolbook inner loop in `mul_schoolbook_ref` does
   `out[i + j]`, `lhs.limbs[i]`, `rhs.limbs[j]` — three checks per
   inner iteration. C++ `std::vector::operator[]` is undefined
   behaviour on out-of-bounds, no check generated. For a `mul_ref`
   that runs k×k checked indices per multiply, this compounds. Most
   likely single contributor; would close if the Rust impl switched
   to `get_unchecked` in the proven-safe inner loops.

2. **Drop / destructor cadence.** Both impls volatile-zero the entire
   allocated capacity on every `BigUint` destruction. The cost is the
   same per instance — but Rust's `Drop` sometimes fires on
   temporaries that C++ collapses via copy-elision / RVO into the
   destination. The 4 KiB-block path is where this matters most (274
   chunks × many transient `BigUint`s per chunk), which is plausibly
   why the ratio peaks at `shamir_split_4kb`.

3. **Allocator path.** macOS libc++ and Rust's `std::alloc::System`
   both eventually hit `malloc`. On paper they look the same. In
   practice libc++ on macOS uses size-class binning slightly more
   aggressively for small buffers, which favours the 1–4 limb
   allocations dominating BigUint workloads. Effect is small per
   allocation, real in aggregate.

4. **Code-gen of the schoolbook inner loop.** Both compile through
   LLVM, but rustc and clang see slightly different IR — Rust's
   `noalias` annotations, panic-unwinding edges from the bounds
   checks, and the slightly different overflow semantics (Rust
   requires `wrapping_*` opt-in; C++ unsigned overflow is defined
   modular wraparound) all influence the inliner and register
   allocator. AppleClang on Apple silicon happens to produce
   slightly tighter code on the Mersenne fold and the schoolbook
   here.

## What this is *not*

- **Not algorithmic.** Same schoolbook, same Karatsuba threshold,
  same Mersenne fold, same Lagrange. Algorithmic equivalence is
  pinned by the cross-language vector tests.
- **Not idiomatic differences.** Both ports use the canonical
  pattern for `BigUint` on a limb vector. There's no clever C++
  trick or sloppy Rust here.
- **Not a constant-time claim difference.** Neither side is
  constant-time; the cpp/include and Rust both document this. The
  bench is wallclock under random operands.

## Reproducing

```bash
# Build both binaries.
cargo build --release --bin pilot_ss
cmake -S cpp -B cpp/build && cmake --build cpp/build -j

# Drive both through pilot-bench.
PILOT_SS_ITERS_PERCENT=100 cpp/scripts/bench_compare.sh
```
